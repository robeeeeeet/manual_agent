# Skillレビュー: `project-testing`

対象: `.claude/skills/project-testing/`

## 良い点

- **フロント/バック/ E2E の導線がある**: Vitest / pytest / Playwright を俯瞰できる。
- **モック例が実務的**: Gemini/Supabaseなど外部依存のモック例があり、テストが書き始めやすい。
- **カバレッジまで言及**: 実行コマンド例があり運用に繋げやすい。

## 懸念点

- **前提の揺れが出やすい**: pytest例にSQLAlchemy等が出る一方で、実際のプロジェクトが「DB層をどう持つか」によってテンプレ適用が変わる。
- **“どれを採用するか”の指針が薄い**: 単体/統合/E2Eの境界や、CIでの最小セットがあると迷いが減る。

## 改善提案（優先度順）

1. **（中）テスト戦略の最小指針を `SKILL.md` に追記**
   - 例: 「PRでは unit+lint、mainでは e2e」など。
2. **（中）pytestテンプレを2系統に分割**
   - DBなし（純サービス/外部モック中心）と、DBあり（統合/fixture）を分ける。
3. **（中）DoD（完了条件）を追加**
   - 例: 代表APIのpytestが通る、代表UIのvitestが通る、最低1本のE2Eが通る、等。
